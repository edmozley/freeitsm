# FreeITSM Changelog

This file tracks all fixes, improvements, and features for website updates.
Items are logged here and then published to the updates page at freeitsm.co.uk/updates.html.

Each entry has a unique ID (3-digit, sequential) used as `data-id` on the website update items.
When publishing to the website, move entries from **Unpublished** to the **Published** section.

| 234 | Project           | Feature     | **Process Mapper translated to all 8 phase-1 languages**. | 233 | Project           | Feature     | **Native multi-language support (i18n) — phase 1**. Lays the foundation for translating the FreeITSM UI into multiple languages, plus a pilot conversion of the Process Mapper module so switching to French produces visible UI changes. **PHP `t()` helper** in new `includes/i18n.php` with namespace-dot-path keys (`t('common.save')` → `lang/{locale}/common.php` key `save`, `t('tickets.status.open')` → `lang/{locale}/tickets.php` nested `status.open`). Resolution falls back **per key** not per file — a 20%-translated `de/tickets.php` gets you 80% German + 20% English fallback for the missing keys, not all-or-nothing. Last resort returns the key itself so unfilled strings are visible during development. **JS `t()` helper** in new `assets/js/i18n.js` mirroring the PHP contract, reading from `window.translations` injected by PHP. The PHP `I18n::exportForJs([$namespaces])` already merges English fallback into the active locale per key so the JS lookup is just a flat dotted walk. **Eight supported locales**: `en` (default), `fr`, `de`, `es`, `pt-BR`, `nl`, `it`, `pl` — BCP 47 codes, matches HTML `lang` attribute. Common UI strings (`save`, `cancel`, `delete`, `add`, `loading`, `yes`, `no`, etc.) translated for all 8; Process Mapper toolbar + autosave + detail panel labels + export modal translated for French as a tangible demo, other 6 locales fall back to English per-key. **Locale resolution chain**: logged-in analyst's `interface_language` user preference → browser `Accept-Language` header (with primary-subtag matching — `pt` resolves to `pt-BR`) → `'en'` default. **System → Preferences** grows an Interface Language dropdown showing each locale in its native name (Français, Deutsch, Español…) — change triggers a save to `user_preferences` (existing API, no new endpoints) then a page reload so PHP re-renders in the new language. **Pilot module conversion**: Process Mapper's toolbar labels, autosave status indicator states (Saved / Unsaved / Saving / Save failed / Autosave off), Mermaid export modal, and key toast messages all now resolve through `t()`. Adding a new language is now strictly a code change: add to `I18n::SUPPORTED_LOCALES`, create `lang/<code>/`, drop in the .php files. |

| 231 | Process Mapper    | Fix         | **Lane divider drag now actually moves the contents below**. Previously the divider followed the cursor pixel-by-pixel but each per-frame `delta` (often 1px) got eaten by `snap()` when applied to step positions — `snap(200 + 1) = 200` — so contents never moved no matter how far the divider was dragged. The divider visually drifted away from anything below it, leaving items glued at their original `y`. Fix: snap `targetH` to the grid in the resize mousemove handler (`const targetH = Math.max(80, snap(laneDragging.startHeight + dy))`), so lane heights now change in 20px increments matching the step grid. With grid-aligned deltas, the shift in `resizeLaneTo` applies cleanly: contents move 20px each time the divider crosses a grid line. Drag feels "snappy" in the same way step drag already does — consistent across the whole UI. Detail panel's Height input intentionally left unsnapped so typed values are respected for fine-tuning. |
| 230 | Process Mapper    | Fix         | **Resizing a lane now physically moves everything below the divider** with it (lanes, steps, groups). Previously the resize shifted steps in lanes below based on `step.lane_id` matching, so any step with a missing or stale `lane_id` (e.g. one placed before its lane was tagged) would stay anchored at its old `y` while its lane band shifted underneath it — leaving the step "stranded" in the wrong lane and creating confusion about which lane it belonged to. Fix swaps the criterion from `lane_id` matching to **visual centre position vs the resized lane's bottom edge** (matching how groups already worked), so anything visually below the divider moves by the same delta the divider moved. Stale `lane_id` values are not re-tagged inside the resize — that's left to the existing auto-tag pass at the start of the next lane drag, which keeps the lane logic itself untouched. |
| 229 | Process Mapper    | Improvement | **Detail panel now survives autosave** — previously, every autosave call ran `openProcess()` after the response to refresh real IDs, and `openProcess()` unconditionally called `closeDetail()`. So if you were editing a step/group/lane's properties, the panel would slide closed every 2 seconds and you'd have to re-click the thing to keep going. Fix has two parts: (1) `openProcess()` gains a `preserveDetail` parameter; `save()` passes `true`, sidebar process-click still passes the default `false` so switching processes still closes the panel cleanly. (2) `save()` captures the current selection by **stable identity** (step's `{x, y}`, group's `{x, y, width, height}`, lane's `display_order`) *before* the save runs, then after `openProcess(newId, true)` reloads the data with fresh real IDs, re-finds the equivalent entity and calls `showDetailForStep/Group/Lane()` to re-populate the panel with the new IDs. Position-based matching is robust because positions are saved as part of the same payload, so the reloaded data has the same coords. If the entity can't be found after reload (e.g. it was deleted in the meantime), the panel closes gracefully. Multi-step selections (size > 1) aren't preserved — the panel only shows single-entity details anyway. |
| 228 | Process Mapper    | Fix         | **Group/step/lane drag occasionally snapped back mid-drag** because the autosave timer (2s after the last `markDirty`) could fire *during* a drag. `save()` calls `openProcess()` after the response lands to refresh real IDs, and that reload destroys the in-progress element and calls `closeDetail()` — so the user saw the panel slide closed and the group/step/lane revert to its last-committed size.
